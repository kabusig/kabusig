## 1. 技術スタック

| 領域 | 技術 |
|---|---|
| フロント/API | Next.js 15(App Router)+ TypeScript + Tailwind CSS 4 |
| ホスティング | Vercel(Pro)。`main` に push で自動デプロイ |
| DB(本番) | Supabase(PostgreSQL + Auth)。無料プラン運用 |
| DB(ローカル) | **node:sqlite**(Node 24標準)。実体 `data/local.db` |
| バッチ | Python 3.13(`.venv`)。pandas等でテクニカル計算 |
| 株価データ | J-Quants API **v2**(`x-api-key` 方式) |
| 通知 | LINE Messaging API(Bot)+ LINE Login(連携) |
| 決済 | Stripe(本番モード・サブスク980円/月)+ Webhook + Billing Portal |
| 認証メール | Resend(独自SMTP、ドメイン kabusig.com 認証済み) |
| 定期実行 | GitHub Actions(日次バッチ)+ Vercel Cron(ニュース) |
| アクセス解析 | Vercel Web Analytics(Cookie不使用) |

**重要**: better-sqlite3 は使わない(このPCにVS C++ビルドツールが無くnode-gyp失敗)。Web側は `node:sqlite` を lazy require。node:sqliteの行はnullプロトタイプなのでClient Componentに渡す前に `{...row}` でプレーン化する(`lib/db.ts` の `plain()/plainOne()`)。

| `NEXT_PUBLIC_APP_URL` | https://kabusig.com |

### ローカル `apps/web/.env.local`(batch も python-dotenv でここを読む)
- 上記のうち開発に必要なもの + `JQUANTS_API_KEY`(J-Quants) + `SQLITE_PATH`/`DB_PATH`(SQLite実体)。
- Supabase未設定なら Web は**開発モード**(`lib/auth.ts` の devMode = 全機能閲覧可)。

### GitHub Actions Secrets / Variables(日次バッチ用)
- Secrets: `JQUANTS_API_KEY`(値の入れ間違いで403になるので厳密に), `SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SUPABASE_URL`, `LINE_CHANNEL_ACCESS_TOKEN`
- Variables: `SEND_LINE`(`true` でLINE通知を送信)

## 4. データ層と切替

- `apps/web/lib/data.ts` が **SQLite(ローカル)/ Supabase(本番)を自動切替**(Supabase env の有無で判定)。
- 本番読み取りは `data-supabase.ts`(PostgREST。ビュー `v_signal_events` / `v_stocks_with_price` / `v_signal_stats` / `v_signal_daily_counts` を使用)。
- ローカルは `db.ts`(node:sqlite)。

## 5. DBスキーマ(要点)

- マイグレーション: `supabase/migrations/0001_init.sql`(テーブル・RLS)、`0002_views_auth.sql`(ビュー・プロフィール自動生成トリガ・watchlist 50件制限)、`0003_notification_all_stocks.sql`(通知の全銘柄フラグ)。
- 主要テーブル: `stocks`, `daily_prices`, `daily_indicators`, `signal_types`, `signal_events`(return_1d/2d/3d 実績を保持), `signal_stats`(事前計算統計), `calendar_events`, `news_links`, `profiles`(plan: free/paid, stripe_*), `watchlists`, `notification_settings`(enabled, **all_stocks**), `notification_logs`(冪等)。
- **新しいマイグレーションを足したら、Supabaseの SQL Editor で手動実行が必要**(自動適用していない)。

## 6. バッチ(Python `batch/`)

| ファイル | 役割 |
|---|---|
| `fetch_stocklist.py` | 東証プライム銘柄マスタ(J-Quants、JPX Excelフォールバック) |

| `calc_indicators.py` / `indicators.py` | テクニカル指標計算 |

| `anomalies.py` | 暦アノマリー(満月・新月は削除済み) |
| `calc_signal_results.py` | 検知後1/2/3営業日の騰落率 + `signal_stats` 再構築(histogram, recent_occurrences) |
| `push_supabase.py` | SQLite→Supabase同期(保持期間: 価格400日/イベント300日で prune) |

- **日次自動実行**: `.github/workflows/batch-daily.yml`(cron `0 9 * * 1-5` = 平日18時JST)。SQLiteを actions/cache で持ち回り、初回のみ `--bulk`、以降 `--date` 増分。`SEND_LINE=='true'` で通知。

## 7. 認証・決済・通知の要点

- **認証**: Supabase Auth。パスワード + マジックリンク併用(`signInWithOtp`)。PKCEフロー(`@supabase/ssr`)。認証メールはResend独自SMTP。テンプレートはSupabaseダッシュボードで日本語化(Magic Link / Confirm signup 両方)。
- **決済**: `api/checkout`(サブスクセッション作成→Stripeへ)、`api/stripe-webhook`(3イベント: `checkout.session.completed` で paid化、`customer.subscription.updated/deleted` で free化)、`api/portal`(Billing Portal。本番はStripe側でポータル有効化が必要)。checkout は try/catch でエラーを `/subscribe?error=` に表示。
- **LINE**: `api/line/login` と `callback`。redirect_uri は `request.headers.get("host")` から生成(Vercel内部URLと不一致を避けるため)。LINE Loginチャネルは公開済み。

## 8. 【最重要】法的制約 — 禁止ワード機械テスト

- サービスは**投資助言・勧誘に該当してはいけない**(金融商品取引法)。
- `batch/tests/test_forbidden_words.py` が**全ソース(コメント・README・.md含む)を走査**し、NG表現があればテスト失敗。**コード変更・文言追加時は必ずこのテストを通す**。
- 禁止語の実体はそのファイル内の `FORBIDDEN` リスト参照。UI文言は「状態の検知」に徹し、売買推奨・利益示唆を書かない。
- 全通知・ランキング等に免責文(`constants.ts` の `DISCLAIMER` / `BACKTEST_NOTE`)を添える。
- ドキュメント類(この指示書等)は `EXCLUDE_NAMES` で除外している。

## 10. 既知の制約・注意点

- Supabase無料枠(500MB)を超えると書込停止 → 保持期間prune(`push_supabase.py`)で対策済み。データを増やすなら容量に注意。
- LINE無料枠は月200通程度 → 通知は監視銘柄 or 発生の少ないシグナルの「全銘柄」向け。高頻度シグナルを全銘柄通知にすると枠を超える(UIで警告済み)。
- Stripeのテスト/本番でオブジェクトID(price/whsec)は別物。切替時は必ず本番モードでコピー。
- ブラウザプレビューのスクリーンショットがこの環境で不安定(DOM/curlで代替検証することがある)。
