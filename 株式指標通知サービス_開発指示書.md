# 株式指標通知サービス 開発指示書

Claude Code への実装指示書。この文書を読んだ上で、フェーズ順に実装すること。

## 0. プロジェクト概要

### コンセプト
日本株（東証銘柄）のテクニカル指標・決算スケジュールを**機械的に検知して通知する**Webアプリ + LINE通知サービス。

**これは投資助言サービスではない。** 「買い時・売り時」を教えるのではなく、「指標が条件に一致した」「決算発表が近い」という**客観的事実のみ**を通知する情報ツールである。この立て付けは法的リスク回避のための本質的な設計思想であり、全実装・全文言でこれを厳守すること。

### ターゲット
- 第一義的には開発者本人（自分用ツール）
- 公開し、他ユーザーも利用可能。有料会員機能あり

### 主要機能（3本柱）
1. **指標ダッシュボード** — RSI/移動平均/出来高等の可視化とシグナル検知履歴
2. **バックテスト** — 指標条件の過去成績を統計表示
3. **ニュースリンク集** — 外部記事へのリンク集（要約はせずリンク誘導が基本、無料公開）

+ **LINE通知**（シグナル検知・決算予定の通知。有料会員の中核価値）

## 1. 【最重要】法的制約と表現ルール

### 1-1. 大原則
金融商品取引法上の「投資助言・代理業」（無登録営業は5年以下の懲役もしくは500万円以下の罰金）に該当しないよう、以下を厳守する。

**サービスが行ってよいこと:**
- テクニカル指標の数値・状態を機械的に計算し表示・通知すること
- 決算発表予定日などの公開スケジュール情報の通知
- 過去データに基づく統計的事実の表示（バックテスト結果）
- 外部ニュース記事へのリンク掲載

**サービスが行ってはならないこと:**
- 特定銘柄の売買を推奨する表現（明示・黙示問わず）
- ユーザー個別の状況に応じた個別・相対性の高い投資判断の提供
- 「おすすめ銘柄」「注目銘柄」等のランキング・キュレーション
- 将来の株価予想の断定的表現

### 1-2. 文言ルール（UI・通知文・全テキストに適用）

| 禁止表現 (NG) | 使用する表現 (OK) |
|---|---|
| 買い時です / 売り時です | 条件に一致しました / シグナルを検知しました |
| 〜しましょう / 〜すべき | 〜の状態です / 〜が発生しました |
| おすすめ / 推奨 / 注目 | 監視中 / 検知履歴 |
| 上がりそう / 下がりそう | 過去N回中M回、翌営業日に上昇（統計的事実のみ） |
| 買い / 売り（ボタン等） | 詳細を見る / チャートを見る |

- 全通知文の末尾に固定文言: `※本通知は情報提供であり、投資勧誘・投資助言ではありません。投資判断はご自身の責任で行ってください。`
- Webアプリの全ページフッターに免責文言を常時表示
- 銘柄一覧の並び順は「証券コード順」「シグナル検知時刻順」等の中立的順序のみ。「おすすめ順」は実装禁止
- シグナル名は「買いシグナル/売りシグナル」ではなく指標の状態名を使う（例:「RSI過熱」「RSI売られすぎ水域」「ゴールデンクロス」「デッドクロス」「出来高急増」）
  - 注: 「ゴールデンクロス」等は一般的なテクニカル用語なので使用可。ただし「=買い」と誘導する説明文は付けない。用語解説は中立的な定義のみ（例:「短期移動平均線が長期移動平均線を上から下に抜けた状態」）

### 1-3. コードレビュー観点
実装完了ごとに、生成した全UI文言・通知テンプレートを grep し、NG表現（買い時, 売り時, おすすめ, 推奨, 注目銘柄, すべき, しましょう）が含まれないことを確認するテストを書くこと。

```
# 例: tests/test_forbidden_words.py
FORBIDDEN = ["買い時", "売り時", "おすすめ", "推奨", "注目銘柄", "買うべき", "売るべき", "買いましょう", "売りましょう"]
# templates/, src/ 配下の全 .py .html .tsx .ts ファイルを走査して検出したら fail
```

## 3. データソース仕様

### フェーズ1（プロトタイプ）: yfinance
- ティッカーは `{証券コード}.T`（例: `7203.T`）
- 日足の四本値・出来高を取得。個人利用・検証目的
- **注意**: yfinanceは非公式ライブラリで商用利用規約がグレー。公開サービス化の前にフェーズ2へ移行すること

### フェーズ2（公開運用）: J-Quants API（東証グループ公式）
- https://jpx-jquants.com/ でアカウント登録、リフレッシュトークン取得
- 無料プランの制約:
  - データは**12週間遅延**、株価履歴は2年分
  - APIコール数 **5件/分** 制限 → バッチはレートリミット(13秒間隔以上)を実装すること
  - **決算発表予定日 API は無料プランで利用可**（遅延の影響を受けにくいスケジュール情報）
- リアルタイム性が必要な株価は有料プラン（Light以上）へ課金して解消
- 適時開示のリアルタイム取得が必要になったら TDnet-addon（有料アドオン、開示から数分〜数十分遅延で取得可）を契約

### データ取得ポリシー
- スクレイピング（株探・Yahoo!ファイナンスHTML等）は実装禁止（規約違反リスク）
- ニュースリンク集のデータソースは RSS 公式配信のみ利用（例: 各メディアの公式RSS）。RSSが無いメディアは手動登録運用

## 4. データベース設計（Supabase / PostgreSQL）

```sql
-- ユーザー拡張情報（Supabase auth.users と 1:1）
create table profiles (
  id uuid primary key references auth.users,
  line_user_id text unique,          -- LINE連携後に格納
  plan text not null default 'free', -- 'free' | 'paid'
  stripe_customer_id text,
  stripe_subscription_id text,
  created_at timestamptz default now()
);

-- 銘柄マスタ
create table stocks (
  code text primary key,             -- '7203'
  name text not null,                -- 'トヨタ自動車'
  market text,                       -- 'プライム' 等
  updated_at timestamptz default now()
);

-- 日足株価
create table daily_prices (
  code text references stocks,
  date date,
  open numeric, high numeric, low numeric, close numeric,
  volume bigint,
  primary key (code, date)
);

-- 計算済み指標（日次）
create table daily_indicators (
  code text references stocks,
  date date,
  rsi14 numeric,
  sma5 numeric, sma25 numeric, sma75 numeric,
  volume_ratio20 numeric,            -- 出来高 / 過去20日平均
  macd numeric, macd_signal numeric,
  primary key (code, date)
);

-- シグナル定義（マスタ）
create table signal_types (
  id text primary key,               -- 'rsi_overbought' 等
  name text not null,                -- 'RSI過熱水域入り'
  description text not null,         -- 中立的な定義文
  is_premium boolean default false   -- 有料会員限定シグナルか
);

-- シグナル検知履歴
create table signal_events (
  id bigint generated always as identity primary key,
  code text references stocks,
  signal_type text references signal_types,
  date date not null,
  detail jsonb,                      -- {"rsi": 72.3, "threshold": 70} 等
  created_at timestamptz default now(),
  unique (code, signal_type, date)
);

-- 決算発表予定
create table earnings_schedule (
  code text references stocks,
  announce_date date,
  fiscal_quarter text,               -- '2026Q1' 等
  primary key (code, announce_date)
);

-- ユーザーの監視銘柄
create table watchlists (
  user_id uuid references profiles,
  code text references stocks,
  created_at timestamptz default now(),
  primary key (user_id, code)
);

-- ユーザーの通知設定
create table notification_settings (
  user_id uuid references profiles,
  signal_type text references signal_types,
  enabled boolean default true,
  custom_threshold numeric,          -- 有料会員のみ閾値カスタム可
  primary key (user_id, signal_type)
);

-- 通知送信ログ（重複送信防止・監査用）
create table notification_logs (
  id bigint generated always as identity primary key,
  user_id uuid references profiles,
  signal_event_id bigint references signal_events,
  channel text default 'line',
  sent_at timestamptz default now(),
  unique (user_id, signal_event_id)
);

-- ニュースリンク
create table news_links (
  id bigint generated always as identity primary key,
  title text not null,
  url text not null unique,
  source_name text not null,         -- 媒体名
  published_at timestamptz,
  tags text[],                       -- ['決算', '日銀'] 等
  created_at timestamptz default now()
);
```

- 全テーブルに Row Level Security を設定。ユーザーデータ（watchlists 等）は本人のみ読み書き可。マスタ系は全員読み取り可・書き込みはservice roleのみ

## 5. シグナル仕様（初期実装セット）

判定は日足ベース。バッチ `detect_signals.py` が大引け後に実行。

| signal_type ID | 名称 | 判定条件 | 無料/有料 |
|---|---|---|---|
| rsi_overbought | RSI過熱水域入り | RSI(14)が前日≤70かつ当日>70 | 無料 |
| rsi_oversold | RSI売られすぎ水域入り | RSI(14)が前日≥30かつ当日<30 | 無料 |
| golden_cross | ゴールデンクロス | SMA5が前日≤SMA25かつ当日>SMA25 | 有料 |
| dead_cross | デッドクロス | SMA5が前日≥SMA25かつ当日<SMA25 | 有料 |
| volume_surge | 出来高急増 | 当日出来高が過去20日平均の3倍以上 | 有料 |
| price_limit_high | 年初来高値更新 | 当日高値が過去1年の高値を更新 | 有料 |
| earnings_upcoming | 決算発表前通知 | 決算発表予定日の2営業日前 | 無料(1日1回まとめ) / 有料(銘柄別即時) |

- 「前日クロス」判定にすることで同一シグナルの連日重複通知を防ぐ
- 有料会員は `custom_threshold` でRSI閾値等を変更可能（65/75等）

## 6. LINE通知仕様

### 構成
1. LINE Developers で Messaging API チャネルを作成（LINE公式アカウント）
2. ユーザーはWebアプリから「LINE連携」ボタン → LINEログイン(OAuth) → `line_user_id` を profiles に保存
3. バッチが signal_events と watchlists を突合し、Push Message API で送信

### 通知テンプレート（この文面を厳守）
```
【シグナル検知】{code} {name}
条件: {signal_description}
検知値: {detail}（例: RSI(14) = 72.3）
終値: {close}円 / 前日比 {change_pct}%
チャート: {app_url}/stocks/{code}

※本通知は情報提供であり、投資勧誘・投資助言ではありません。投資判断はご自身の責任で行ってください。
```

```
【決算発表予定】
{date} に決算発表が予定されている監視銘柄:
・{code} {name}（{fiscal_quarter}）
詳細: {app_url}/stocks/{code}

※本通知は情報提供であり、投資勧誘・投資助言ではありません。
```

### 制約・実装注意
- LINE Messaging API の無料メッセージ枠は月200通（2026年時点の無料プラン）。超過見込みになったらLINE側プラン増強が必要。通知ログから月間送信数を集計するダッシュボードを管理画面に作ること
- 無料会員: 1日1回、17:00 JST に当日分をまとめて1通で送信（メッセージ数節約 + 有料との差別化）
- 有料会員: シグナル検知バッチ実行の都度、銘柄別に送信
- `notification_logs` の unique 制約で重複送信を防止
- 送信失敗（ブロック等）は指数バックオフでリトライ、3回失敗でスキップしログに記録

## 7. 料金プラン・課金仕様

**課金対象は「機能・利便性」であり「情報の中身」ではない**（助言性の否定のため。この整理を pricing ページの説明文にも反映）。

| 機能 | 無料 | 有料（月額 500〜980円想定、Stripeで設定） |
|---|---|---|
| ダッシュボード閲覧 | ◯ | ◯ |
| ニュースリンク集 | ◯ | ◯ |
| 監視銘柄数 | 3 | 50 |
| LINE通知 | 1日1回まとめ | シグナル種別ごと都度通知 |
| シグナル種別 | RSI系 + 決算予定のみ | 全種別 |
| 閾値カスタマイズ | ✕ | ◯ |
| バックテスト | 直近6ヶ月・プリセット条件のみ | 全期間・条件自由設定 |

### Stripe 実装
- Stripe Checkout でサブスク登録 → webhook (`checkout.session.completed`, `customer.subscription.deleted` 等) で profiles.plan を更新
- 解約はStripeカスタマーポータルへリンク（自前実装しない）
- **重要**: 特商法の最終確認画面要件（2022年改正）に対応するため、Checkout 直前の自前確認画面に「月額料金・課金周期・解約方法・提供内容」を明示すること

## 8. バックテスト仕様

### 目的
「このシグナルは過去にどう機能したか」の**統計的事実**を表示する。将来予測ではないことを画面上に明記。

### 計算ロジック（backtest_engine.py）
- 入力: 銘柄コード（または全銘柄）、シグナル種別、対象期間、保有期間N営業日（1/5/20から選択）
- 処理: 過去の signal_events 各発生日について、発生翌営業日始値→N営業日後終値の騰落率を計算
- 出力:
  - 発生回数
  - 上昇回数 / 下落回数（勝率という言葉は使わず「上昇した割合」と表記）
  - 平均騰落率、中央値、最大上昇、最大下落
  - 騰落率の分布ヒストグラム
- 画面には必ず併記: 「過去の統計であり、将来の値動きを保証・示唆するものではありません」

### UI
- 条件フォーム → 結果カード（統計値）+ ヒストグラムチャート + 発生日一覧テーブル
- 無料会員は期間が直近6ヶ月固定・シグナル種別プリセットのみ選択可

## 9. ニュースリンク集仕様

- **要約・論評は行わない。タイトル + 媒体名 + 公開日時 + 外部リンクのみ**（著作権・助言性リスクの両方を回避する設計判断）
- データ投入: 管理画面から手動登録 + 公式RSSの自動取り込みバッチ（1日2回）
- タグ（決算 / 金融政策 / 市況 など）でフィルタ可能
- 将来「発言まとめ」をやる場合も、引用は主従関係・出所明記・必要最小限の引用ルールを守った独自記事とすること（フェーズ3以降、本指示書のスコープ外）

## 10. 法務ページ（4点、実装必須）

以下のドラフトを `app/legal/` 配下に静的ページとして実装。文言は下記をベースに使用してよいが、**公開前に専門家（行政書士・弁護士）レビューを受ける**前提のテンプレートである旨をコード内コメントに残すこと。サービス名・運営者名等は環境変数または定数ファイルで一元管理し、`【サービス名】`等のプレースホルダを置換する。

### 10-1. 免責事項（disclaimer）骨子
- 本サービスは金融商品取引法上の投資助言・代理業に該当する行為を行うものではない
- 提供する情報はテクニカル指標等の機械的計算結果および公開情報であり、特定の金融商品の売買を推奨するものではない
- 投資判断はすべて利用者自身の責任で行うものとし、本サービスの利用により生じた損害について運営者は一切の責任を負わない
- データは遅延・欠損・誤りを含む可能性がある
- 本サービスは予告なく内容の変更・停止を行うことがある

### 10-2. 利用規約（terms）骨子
- 定義 / 適用範囲
- アカウント登録・管理（正確な情報の登録義務、アカウント譲渡禁止）
- 有料プラン（料金、支払方法=クレジットカード/Stripe、課金周期、解約方法、日割返金なし）
- 禁止事項（スクレイピング、リバースエンジニアリング、再配布、自動化アクセス、通知内容の商用転載）
- 免責（10-1 の内容を条文化 + サービス無保証）
- 知的財産権
- 規約変更手続（変更時の通知方法と効力発生）
- 準拠法（日本法）・合意管轄（運営者所在地の地方裁判所）

### 10-3. プライバシーポリシー（privacy）骨子
- 取得する情報: メールアドレス、LINEユーザーID（連携時）、決済情報（Stripe管理でありサービスはカード番号を保持しない）、利用ログ
- 利用目的: サービス提供、通知送信、課金管理、問い合わせ対応、不正利用防止
- 第三者提供: 法令に基づく場合を除き行わない。委託先（Stripe, Supabase, LINE, Vercel）への取扱い委託の明記
- 保存期間・削除請求・開示請求への対応窓口（メールアドレス）
- Cookie・アクセス解析を使う場合はその旨

### 10-4. 特定商取引法に基づく表記（tokushoho）
**注意事項（実装者へ）:**
- 個人事業主の場合、**氏名（戸籍上の氏名）は原則として省略不可**（屋号のみの表記は不可という見解が主流）。氏名は記載する前提で作ること
- **住所・電話番号**は「請求があれば遅滞なく開示する」旨を表示し、実際に遅滞なく開示できる体制があれば省略可（特商法11条ただし書）
- バーチャルオフィス住所の利用も選択肢（消費者庁は一定条件下で認める見解）

記載項目テンプレート:
```
販売事業者名: 【氏名】（屋号: 【屋号】）
所在地: 請求があれば遅滞なく開示いたします（特定商取引法第11条ただし書に基づき省略）
電話番号: 請求があれば遅滞なく開示いたします（同上）
```

## 13. 実装上の細かい指示

- 指標計算は必ず調整後終値ではなく**生の終値**を使う場合の注意（株式分割時に指標が飛ぶ）を考慮し、yfinance では `auto_adjust=True` の調整済み価格を使用。J-Quants移行時は AdjustmentClose を使用
- 営業日判定は J-Quants の取引カレンダーAPI（無料プランで利用可）を使用。フェーズ1では `jpholiday` ライブラリ + 土日除外で代替
- バッチの冪等性: 同日再実行してもシグナル・通知が重複しないこと（unique制約 + upsert）
- タイムゾーンは全て JST（Asia/Tokyo）で統一。DBはUTC保存・表示時にJST変換
- テスト: 指標計算（既知の値との突合）、シグナル判定（境界値）、禁止ワード、の3種は必須
- コミットは機能単位で細かく。フェーズ完了ごとにタグを打つ

以上。不明点があれば実装前に確認すること。
